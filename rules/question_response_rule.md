# Правило ответа на вопросы пользователя

## Описание
Когда пользователь задает вопрос, AI должен:
1. Дать прямой и понятный ответ на вопрос
2. Предложить варианты решения (если применимо)
3. Спросить уточняющие вопросы, если что-то неясно

## Формат ответа

### 1. Прямой ответ
Сначала дайте четкий ответ на вопрос пользователя.

### 2. Варианты решения
Если вопрос подразумевает несколько способов решения, предложите варианты в формате:

**Вариант 1:** [Описание]
- Плюсы: ...
- Минусы: ...
- Когда использовать: ...

**Вариант 2:** [Описание]
- Плюсы: ...
- Минусы: ...
- Когда использовать: ...

**Вариант 3:** [Описание]
- Плюсы: ...
- Минусы: ...
- Когда использовать: ...

### 3. Уточняющие вопросы
Если что-то неясно, задайте вопросы:
- "Уточните, пожалуйста: ..."
- "Какой вариант вам больше подходит?"
- "Нужна ли дополнительная информация о ...?"

## Примеры

### Пример 1: Технический вопрос
**Вопрос:** "Как лучше организовать хранение файлов?"

**Ответ:**
Хранение файлов можно организовать несколькими способами:

**Вариант 1:** Локальное хранение в `/var/uploads`
- Плюсы: простота, быстрый доступ
- Минусы: проблемы с масштабированием, нет резервного копирования
- Когда использовать: для небольших проектов, MVP

**Вариант 2:** Облачное хранилище (S3, Yandex Object Storage)
- Плюсы: масштабируемость, резервное копирование, CDN
- Минусы: дополнительные затраты, зависимость от внешнего сервиса
- Когда использовать: для production, больших проектов

**Вариант 3:** Гибридный подход
- Плюсы: баланс между простотой и масштабируемостью
- Минусы: более сложная архитектура
- Когда использовать: при постепенном росте проекта

Какой вариант вам больше подходит? Учитываете ли вы будущее масштабирование?

### Пример 2: Вопрос о функциональности
**Вопрос:** "Нужно ли добавлять кэширование?"

**Ответ:**
Кэширование стоит добавить, если:
- Частые запросы к БД замедляют работу
- Есть данные, которые редко меняются
- Нужно снизить нагрузку на сервер

**Вариант 1:** In-memory кэш (Redis)
- Плюсы: очень быстро, простое использование
- Минусы: данные теряются при перезапуске
- Когда использовать: для временных данных, сессий

**Вариант 2:** Кэш в БД (PostgreSQL)
- Плюсы: персистентность, надежность
- Минусы: медленнее, чем Redis
- Когда использовать: для критичных данных

**Вариант 3:** HTTP кэш (Nginx)
- Плюсы: снижает нагрузку на backend
- Минусы: только для статических/полустатических данных
- Когда использовать: для API ответов, которые редко меняются

Какой тип данных вы планируете кэшировать? Насколько критична скорость доступа?

## Важные моменты

1. **Всегда отвечайте на вопрос** - даже если предлагаете варианты, сначала дайте прямой ответ
2. **Предлагайте 2-3 варианта** - не перегружайте пользователя выбором
3. **Объясняйте контекст** - когда и почему использовать каждый вариант
4. **Задавайте уточняющие вопросы** - если нужно больше информации для точного ответа
5. **Учитывайте правила проекта** - варианты должны соответствовать архитектуре и стандартам проекта

## Исключения

Если вопрос очень простой и не требует вариантов (например, "Какой пароль?"), просто дайте прямой ответ без вариантов.

Если вопрос требует немедленного действия без обсуждения (например, "Исправь ошибку"), выполните действие и объясните, что было сделано.
